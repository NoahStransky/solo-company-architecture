## 技术选型及理由

| 组件 | 选型 | 理由 |
|------|------|------|
| 订阅存储 | SQLite | 零运维，GitHub Actions 原生支持 Python sqlite3 |
| 邮件发送 | Resend API | 免费 3000 封/月，代码简单，支持 HTML |
| 订阅表单 | GitHub Issues + static form | GitHub Pages 只能托管静态页面，用 Issues API 接收订阅请求 |
| 验证机制 | 邮件验证链接（JWT token） | 防止虚假邮箱，符合 CAN-SPAM |
| 邮件模板 | Jinja2 HTML | 与现有博客模板一致，暗色主题 |

## 数据模型

```sql
-- subscribers 表
CREATE TABLE subscribers (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    email TEXT UNIQUE NOT NULL,
    verified BOOLEAN DEFAULT 0,
    verification_token TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    verified_at TIMESTAMP,
    unsubscribed BOOLEAN DEFAULT 0
);

-- send_log 表（用于追踪发送状态）
CREATE TABLE send_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    email TEXT NOT NULL,
    sent_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status TEXT,  -- 'sent', 'bounced', 'failed'
    error_message TEXT
);
```

## API / 接口设计

### 1. 订阅提交（GitHub Issues）
用户填写表单 → 创建 GitHub Issue（带 label: `newsletter-subscription`）

### 2. 验证邮件发送
GitHub Actions 检测到新 Issue → 发送验证邮件 → 更新 Issue 状态

### 3. 验证链接处理
用户点击验证链接 → GitHub Actions workflow dispatch → 更新数据库

### 4. 每日发送
定时触发 → 查询 verified subscribers → 生成邮件 → 批量发送

### 5. 退订
邮件底部链接 → 标记 unsubscribed = 1

## 风险评估

| 风险 | 严重度 | 缓解方案 |
|------|--------|---------|
| GitHub Pages 纯静态，无法直接接收 POST | 🔴 High | 使用 GitHub Issues API 作为"表单后端"，或改用第三方 form 服务 |
| Resend 免费额度 3000/月 | 🟡 Med | 初期够用，超过后升级 $20/月 可发 5 万封 |
| 邮件进垃圾箱 | 🟡 Med | 使用 Resend（自带域名认证），邮件内容避免垃圾词 |
| GitHub Actions 并发限制 | 🟢 Low | 邮件发送串行化，每封间隔 1 秒 |

## 关键决策

1. **不用传统邮件服务（Mailchimp/SendGrid）**：
   - 原因：需要额外账户管理，API 复杂
   - Resend 更轻量，适合开发者

2. **不用服务器/数据库**：
   - 原因：一人公司尽量减少运维
   - SQLite + GitHub Actions 完全托管

3. **验证邮件必须**：
   - 原因：法律合规（CAN-SPAM/反垃圾邮件法）
   - 防止机器人批量注册
