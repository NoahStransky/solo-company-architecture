# 🏗️ 产品开发 SOP — Solo Company Edition

> 一人公司的产品开发流程：从想法到上线的最短路径

## 🔄 完整流程图

```
想法 (Idea)
   │
   ▼
验证 (Validation) ──→ 用户访谈 → 竞品分析 → 市场调研
   │
   ▼
PRD (Product Doc) ──→ 用户故事 → 功能范围 → 验收标准
   │
   ▼
设计 (Design) ──────→ 低保真原型 → 用户流程 → 信息架构
   │
   ▼
技术评估 (Tech Review) ──→ CTO 评估可行性 → 工作量估算
   │
   ▼
开发 (Development) ──→ 编码 → 自测 → 代码审查
   │
   ▼
测试 (Testing) ──────→ 功能测试 → 回归测试 → 性能检查
   │
   ▼
发布 (Release) ──────→ 部署 → 监控 → 用户通知
   │
   ▼
复盘 (Review) ───────→ 数据回顾 → 用户反馈 → 迭代计划
```

---

## 📋 各阶段详细操作

### Phase 1: 验证（1-3 天）

**目标**：确认这个问题值得解决，有人愿意为此付费。

**检查清单**：
- [ ] 我是否和至少 3 个目标用户聊过？
- [ ] 他们是否表现出"痛苦的表情"描述这个问题？
- [ ] 现有解决方案为什么不够好？
- [ ] 如果产品明天上线，他们会立即使用吗？
- [ ] 他们愿意为此付多少钱？

**产出物**：
```markdown
## 验证报告：AI 新闻聚合

### 访谈摘要
- Alex (CTO, 32岁): "每天花 1 小时刷新闻，效率低"
- Lisa (独立开发者, 28岁): "希望有个摘要，通勤时看"

### 痛点确认
- 信息过载（8/10 分）
- 缺乏深度解读（7/10 分）

### 竞品不足
- TechCrunch 太泛
- HN 没有中文

### 付费意愿
- 10/10 受访者愿意付费 ¥20-50/月
```

---

### Phase 2: PRD（0.5-1 天）

**目标**：把模糊的想法变成可执行的开发任务。

**模板**：见 [CPO 职责文档](../org-chart/03-cpo.md#1-产品需求文档prd模板)

**关键原则**：
- 一个 PRD 对应一个可发布的功能
- PRD 必须有"验收标准"（Definition of Done）
- PRD 必须先给 CTO 技术评估，再进入开发

---

### Phase 3: 设计（0.5-1 天）

**目标**：确认用户体验路径，避免开发返工。

**一人公司设计原则**：
1. **先画纸上草图**（5分钟），再上工具
2. **不要追求像素完美**——开发时还会改
3. **重点关注用户流程**，而非视觉细节

**设计检查清单**：
- [ ] 用户从入口到核心价值的步骤 < 3 步
- [ ] 每个页面都有明确的"下一步行动"
- [ ] 错误状态有处理（空数据、加载失败）
- [ ] 移动端可用（即使先桌面端）

---

### Phase 4: 技术评估（与 PRD 并行）

**CTO 在此阶段必须回答**：
1. 这个功能技术上可行吗？
2. 大概需要多少开发时间？
3. 有什么技术风险？
4. 需要引入新的依赖/服务吗？成本多少？

**时间估算方法**：
```
乐观估计 × 1 + 最可能 × 2 + 悲观估计 × 1
───────────────────────────────────── = 建议工期
                4

然后 × 1.5（一人公司上下文切换成本）
```

---

### Phase 5: 开发（核心阶段）

**分支管理**：
```bash
# 功能分支
git checkout -b feature/007-ai-news-summary

# 开发完成，自测后合并
git checkout main
git merge feature/007-ai-news-summary
```

**开发检查清单**：
- [ ] 代码符合团队规范（Black, Ruff, TypeScript strict）
- [ ] 写了基本的单元测试（核心逻辑）
- [ ] 更新了 README / API 文档
- [ ] 本地测试通过
- [ ] 没有 console.log / 调试代码残留

---

### Phase 6: 测试（必须做，但可以简化）

**一人公司测试策略**：

| 测试类型 | 必须程度 | 方法 |
|---------|---------|------|
| 单元测试 | ⭐⭐⭐ 核心逻辑必须 | pytest / vitest |
| 集成测试 | ⭐⭐ 关键流程 | 手动 + 简单脚本 |
| E2E 测试 | ⭐ MVP 阶段可跳过 | Playwright（V2 再加） |
| 性能测试 | ⭐⭐ 上线前检查 | Lighthouse / curl 计时 |
| 安全测试 | ⭐⭐⭐ 永远不要跳过 | 依赖扫描、输入验证 |

**发布前必做（5 分钟）**：
```bash
# 1. 运行测试
pytest

# 2. 检查关键路径手动走一遍
# 3. 检查错误处理（断网、输错数据）
# 4. 手机端打开看看
```

---

### Phase 7: 发布（部署）

**发布检查清单**：
- [ ] 数据库迁移脚本已准备（如有）
- [ ] 环境变量已配置
- [ ] 回滚方案确认（上一个版本可立即切回）
- [ ] 监控告警已开启
- [ ] 用户通知已准备（ changelog / 推文）

**发布策略**：
- **小规模发布**：先自己用 + 给 3 个朋友测试
- **灰度发布**：如果有多用户，先给 10% 用户
- **全量发布**：观察 30 分钟无异常后再推广

---

### Phase 8: 复盘（1 周内）

**数据指标**：
- 功能使用率（多少人用了新功能）
- 错误率（Sentry 告警数量）
- 性能影响（加载速度变化）
- 用户反馈（主动收集）

**复盘模板**：
```markdown
## 功能发布复盘：AI 新闻摘要

### 数据
- 使用率：45%（目标 50%）
- Bug 数：2 个（目标 0）
- 平均加载时间：+200ms（需优化）

### 做得好的
- 用户反馈摘要质量高

### 做得不好的
- 移动端显示有问题
- 加载比预期慢

### 下次改进
- 开发阶段加移动端测试
- 摘要生成加缓存
```

## ⏱️ 时间参考（一人公司）

| 功能规模 | 总时间 | 各阶段分配 |
|---------|--------|-----------|
| 小功能（改文案、加按钮） | 2-4h | 设计 10% + 开发 70% + 测试 20% |
| 中功能（新页面、新 API） | 2-5 天 | 验证 10% + PRD 10% + 设计 10% + 开发 50% + 测试 20% |
| 大功能（新模块、AI 集成） | 1-3 周 | 验证 15% + PRD 10% + 设计 10% + 技术评估 5% + 开发 40% + 测试 20% |

---

## 🚫 禁止事项

1. **没有 PRD 直接写代码**
   - 结果：写一半发现方向错了

2. **没有测试直接上线**
   - 结果：凌晨 3 点修 Bug

3. **同时开发多个大功能**
   - 结果：都做到 80%，没有一个能发布

4. **发布后不管了**
   - 结果：用户遇到问题没人理
